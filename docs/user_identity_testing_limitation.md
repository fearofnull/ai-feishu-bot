# 用户身份测试的限制与解决方案

## 问题说明

飞书 MCP 工具使用的是**应用身份**（tenant_access_token），而不是**用户身份**（user_access_token）。这意味着：

- ✅ 可以：以应用身份发送消息（显示为机器人发送）
- ❌ 不可以：以用户身份发送消息（显示为用户发送）

## 为什么需要用户身份测试？

测试机器人时，我们希望模拟真实用户的操作：
1. 用户发送消息给机器人
2. 机器人接收并处理消息
3. 机器人回复用户

如果使用应用身份发送测试消息，机器人可能不会响应（因为是自己发给自己）。

## 解决方案

### 方案 1：手动测试（推荐）

**步骤**：
1. 在飞书客户端打开"机器人群"
2. 找到要引用的卡片消息
3. 点击"回复"按钮
4. 输入 `@gemini-cli 请分析这条卡片消息`
5. 发送

**优点**：
- 完全模拟真实用户操作
- 可以验证飞书客户端的显示效果
- 简单直接

**缺点**：
- 需要手动操作
- 不适合自动化测试

### 方案 2：使用测试脚本发送，手动验证

**步骤**：
1. 运行脚本发送测试卡片：
   ```bash
   python test_scripts/quick_reply_test.py
   ```

2. 脚本会输出卡片消息的 message_id

3. 你在飞书客户端手动引用该卡片并发送命令

**优点**：
- 半自动化
- 可以快速生成测试数据
- 保留手动验证的准确性

**缺点**：
- 仍需手动操作引用步骤

### 方案 3：使用 Playwright 自动化（高级）

使用浏览器自动化工具模拟用户操作。

**示例代码**：
```python
from playwright.sync_api import sync_playwright

def test_with_user_identity():
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=False)
        page = browser.new_page()
        
        # 1. 登录飞书网页版
        page.goto("https://feishu.cn")
        # ... 登录流程
        
        # 2. 打开目标群聊
        page.goto("https://feishu.cn/messenger/...")
        
        # 3. 找到要引用的消息
        message = page.locator(f'[data-message-id="{message_id}"]')
        
        # 4. 点击回复按钮
        message.hover()
        page.click('button[aria-label="回复"]')
        
        # 5. 输入命令
        page.fill('textarea', '@gemini-cli 请分析这条卡片消息')
        
        # 6. 发送
        page.click('button[aria-label="发送"]')
        
        browser.close()
```

**优点**：
- 完全自动化
- 真实模拟用户操作
- 可以集成到 CI/CD

**缺点**：
- 实现复杂
- 需要处理登录认证
- 可能受飞书网页版更新影响

### 方案 4：使用应用身份 + 模拟用户（当前方案）

虽然消息显示为应用发送，但可以用于测试机器人的核心功能。

**使用场景**：
- 测试消息解析逻辑
- 测试卡片内容提取
- 测试 AI 执行器
- 测试响应格式化

**限制**：
- 无法测试用户权限相关功能
- 无法测试用户会话管理
- 消息显示为应用发送

## 当前可用的自动化测试

### 1. 快速引用测试
```bash
python test_scripts/quick_reply_test.py
```

**功能**：
- 发送测试卡片消息
- 使用 Reply API 引用该消息
- 输出消息 ID 供验证

### 2. 完整自动化测试
```bash
python test_scripts/automated_reply_test.py
```

**功能**：
- 发送测试卡片
- 引用并发送命令
- 等待机器人响应
- 验证响应内容

### 3. 演示脚本
```bash
python demo_reply_message.py
```

**功能**：
- 演示 Reply API 的完整使用流程
- 验证消息结构
- 输出详细说明

## 推荐的测试流程

### 开发阶段
1. 使用 `quick_reply_test.py` 快速生成测试数据
2. 手动在飞书客户端验证功能
3. 查看机器人日志确认处理流程

### 回归测试
1. 使用 `automated_reply_test.py` 自动化测试
2. 检查测试输出和日志
3. 定期手动验证关键功能

### 生产验证
1. 手动测试真实用户场景
2. 监控机器人日志
3. 收集用户反馈

## 未来改进方向

1. **OAuth 2.0 集成**：
   - 实现用户授权流程
   - 获取 user_access_token
   - 以用户身份发送消息

2. **测试用户账号**：
   - 创建专门的测试账号
   - 使用测试账号进行自动化测试
   - 避免影响真实用户

3. **Webhook 模拟**：
   - 模拟飞书发送的 Webhook 事件
   - 直接测试机器人的事件处理逻辑
   - 绕过消息发送限制

## 总结

由于飞书 MCP 工具的限制，目前无法直接以用户身份自动发送消息。推荐的测试方式是：

1. **日常开发**：使用测试脚本 + 手动验证
2. **自动化测试**：测试机器人的核心逻辑（不依赖用户身份）
3. **生产验证**：手动测试真实用户场景

这种组合方式可以在自动化和准确性之间取得平衡。
