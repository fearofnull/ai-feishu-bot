# AI意图分类功能

## 概述

AI意图分类是智能路由系统的核心改进，使用大语言模型的能力来判断用户消息是否需要访问本地代码库（CLI层），而不是依赖硬编码的关键词匹配。

## 为什么需要AI意图分类？

### 传统关键词检测的局限性

**问题示例**：

```
用户: "如何设计一个代码审查流程？"
关键词检测: 包含"代码"→ 路由到CLI层 ❌
实际需求: 讨论流程设计 → 应该使用API层 ✅
```

```
用户: "给我讲讲代码重构的最佳实践"
关键词检测: 包含"代码重构"→ 路由到CLI层 ❌
实际需求: 学习理论知识 → 应该使用API层 ✅
```

```
用户: "帮我生成一个项目结构的示例"
关键词检测: 包含"项目结构"→ 路由到CLI层 ❌
实际需求: 生成示例代码 → 应该使用API层 ✅
```

### AI意图分类的优势

✅ **理解真实意图**：不被关键词误导，理解用户的真实需求
✅ **处理各种表达**：可以处理同义词、口语化表达、复杂句式
✅ **区分场景**：能区分"讨论代码"和"操作代码"的区别
✅ **持续改进**：可以通过优化提示词来提升准确率

## 工作原理

### 1. 分类流程

```
用户消息
    ↓
检查缓存（避免重复调用）
    ↓
调用AI API进行意图分类
    ↓
解析JSON响应
    ↓
返回分类结果（needs_cli, confidence, reason, category）
    ↓
路由到相应的执行层
```

### 2. 分类提示词

系统使用精心设计的提示词来指导AI进行分类：

```
你是一个智能路由助手。请判断用户的问题是否需要访问本地代码库。

需要访问本地代码库（CLI层）的情况：
- 查看、分析、修改现有代码文件
- 读取项目中的具体文件内容
- 分析项目结构或架构
- 执行本地命令或脚本
- 基于现有代码进行重构或优化

不需要访问本地代码库（API层）的情况：
- 一般性问答（如"你是谁"、"什么是Python"）
- 概念解释和理论知识
- 代码生成（不需要查看现有代码）
- 翻译、写作、总结等文本处理
- 算法讲解和示例代码
```

### 3. 响应格式

AI返回结构化的JSON响应：

```json
{
    "needs_cli": true/false,
    "confidence": 0.0-1.0,
    "reason": "判断理由",
    "category": "意图类别"
}
```

### 4. 降级策略

如果AI分类失败（API不可用、解析错误等），系统会自动降级到关键词检测：

```
AI分类尝试
    ↓ (失败)
关键词检测（降级）
    ↓
返回结果
```

## 配置方法

### 1. 启用AI意图分类

在 `.env` 文件中设置：

```bash
# 启用AI意图分类（推荐）
USE_AI_INTENT_CLASSIFICATION=true

# 禁用AI意图分类（使用关键词检测）
USE_AI_INTENT_CLASSIFICATION=false
```

### 2. 选择分类模型

系统会自动选择最合适的API执行器进行意图分类，优先级：

1. **OpenAI** - 通常更快更便宜，推荐用于意图分类
2. **Gemini** - 备选方案
3. **Claude** - 备选方案

**配置建议**：

```bash
# 如果只配置了一个AI服务，系统会自动使用它
OPENAI_API_KEY=your_key_here

# 如果配置了多个，系统会优先使用OpenAI
OPENAI_API_KEY=your_openai_key
CLAUDE_API_KEY=your_claude_key
GEMINI_API_KEY=your_gemini_key
```

### 3. 性能优化

**缓存机制**：

系统会缓存相同消息的分类结果，避免重复调用AI：

```python
# 第一次调用：需要调用AI
"你是谁" → AI分类 → 缓存结果

# 第二次调用：直接使用缓存
"你是谁" → 从缓存读取 → 立即返回
```

**Token优化**：

- 使用简短的提示词（约200 tokens）
- 限制AI响应长度（max_tokens=200）
- 使用低温度参数（temperature=0.1）以获得确定性结果

## 性能影响

### Token消耗

每次AI意图分类大约消耗：

- **输入**: ~150 tokens（提示词 + 用户消息）
- **输出**: ~50 tokens（JSON响应）
- **总计**: ~200 tokens/次

**成本估算**（以OpenAI GPT-4o为例）：

- 输入: $0.0025/1K tokens
- 输出: $0.01/1K tokens
- 单次分类成本: ~$0.0009（约0.006元人民币）

### 响应时间

- **首次分类**: ~1-2秒（需要调用AI）
- **缓存命中**: <10ms（直接返回）
- **对用户体验的影响**: 几乎无感知（分类和执行可以并行）

### 优化建议

1. **启用缓存**（默认启用）：相同消息不重复分类
2. **使用快速模型**：OpenAI通常比Claude和Gemini更快
3. **合理设置超时**：避免分类耗时过长

## 常见问题

### Q1: AI分类会增加多少成本？

**A**: 成本非常低。假设每天处理100条消息，其中50%命中缓存：
- 实际分类次数: 50次
- 每日成本: 50 × $0.0009 = $0.045（约0.3元人民币）
- 每月成本: ~$1.35（约9元人民币）

### Q2: AI分类失败怎么办？

**A**: 系统有完善的降级机制：
1. AI分类失败 → 自动降级到关键词检测
2. 关键词检测失败 → 使用默认层（API层）
3. 用户始终能得到响应，不会因为分类失败而无法使用

### Q3: 如何提高分类准确率？

**A**: 可以通过以下方式优化：
1. **优化提示词**：在 `intent_classifier.py` 中调整 `CLASSIFICATION_PROMPT`
2. **调整置信度阈值**：根据实际情况设置最低置信度要求
3. **收集反馈**：记录分类错误的案例，用于改进提示词

### Q4: 可以使用本地模型吗？

**A**: 可以！只需要：
1. 部署兼容OpenAI API的本地模型（如Ollama）
2. 设置 `OPENAI_BASE_URL` 指向本地服务
3. 系统会自动使用本地模型进行分类

示例：

```bash
# 使用Ollama本地模型
OPENAI_BASE_URL=http://localhost:11434/v1
OPENAI_API_KEY=ollama  # Ollama不需要真实的API key
OPENAI_MODEL=qwen2.5:7b
```

### Q5: 关键词检测还有用吗？

**A**: 有用！关键词检测作为降级方案非常重要：
- 当AI服务不可用时提供保障
- 在网络不稳定时确保系统可用
- 作为AI分类的验证参考
